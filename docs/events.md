# Events â€” reminders and notifications

Reminder system: one-shot or recurring events (birthdays, namedays,
anniversaries, plain reminders, voice notifications, playback schedules)
stored as JSON, edited via the web UI, fired at a scheduled time by a
background scheduler. Notification = buzzer pattern (or a TTS voice clip
for `voice`) + a fullscreen LVGL screen with an encoder "dismiss" button.
A `schedule` event instead starts audio playback (radio station or an SD
file/folder) and surfaces the player screen. Playback schedules are stored
as a start event with an optional linked stop event and shown as one item â€” no toast.
Plus a small
indicator on selected screens showing how many events are still pending
today. A `calendar` event is a **read-only mirror of the phone calendar**
(bulk-synced from the Android app): it never fires and isn't counted â€” it
only feeds the Home-screen calendar widget (see
[Calendar mirror](#calendar-mirror-ev_calendar)).

> The old `alarm` type (radio-at-time + toast) was folded into `schedule`;
> `type_from_str("alarm")` still maps to `EV_SCHEDULE` so legacy
> `events.json` entries load and re-save as `schedule`.

## Data model

`event_t` defined in [components/events/events_service.h](../components/events/events_service.h):

| Field | Type | Range / description |
|---|---|---|
| `id` | `char[9]` | 8 hex chars + `\0`, generated by `esp_random()` on add |
| `playback_id` | `char[9]` | Identifies one playback schedule; shared by its start and optional stop records, empty for other events |
| `type` | enum | `EV_BIRTHDAY` / `EV_NAMEDAY` / `EV_REMINDER` / `EV_ANNIVERSARY` / `EV_VOICE` / `EV_SCHEDULE` / `EV_CALENDAR` |
| `title` | `char[64]` | Text shown on screen and in web list |
| `year`, `month`, `day` | int | Base date. For `yearly` the year is irrelevant (commemorative). |
| `hour`, `minute` | int | Trigger time (local TZ) |
| `recurrence` | enum | `none` / `daily` / `weekly` / `monthly` / `yearly` |
| `enabled` | bool | `false` excludes the event from firing and from the ðŸ”” counter |
| `station` | int | `EV_SCHEDULE` with empty `sound`: 0-based playlist index to start at trigger time, or `EVENT_STATION_STOP` (-1) to stop playback instead |
| `volume` | int | `EV_SCHEDULE` / `EV_VOICE`: 0..100, applied via `settings_set_volume()` at fire time so playback starts at a predictable level |
| `sound` | `char[128]` | `EV_VOICE`: WAV filename under `/voice`. `EV_SCHEDULE`: SD file/folder path relative to the card root (empty â†’ play `station`) |

Maximum `EVENTS_MAX = 200` events (shared between user-managed and
calendar-mirror events). Static array in PSRAM, guarded by a mutex against
concurrent access from scheduler and CRUD.

Plus runtime-only state (RAM, not persisted):

- `s_fired_today[EVENTS_MAX]` â€” "already fired today" flag, prevents
  double-firing. Reset on day rollover and on `events_add()` /
  `events_update()`.
- `s_last_day` â€” month-day from the previous tick, used to detect day
  rollover.

## Persistence â€” `/spiffs/events.json`

Format:

```json
{
  "events": [
    {
      "id": "a3f19c2b",
      "type": "reminder",
      "title": "Pills",
      "year": 2026, "month": 4, "day": 23,
      "hour": 8, "minute": 0,
      "recurrence": "daily",
      "enabled": true
    }
  ]
}
```

Atomic write in `save_to_file()`: write to `events.json.tmp`, then
`remove(events.json)` + `rename(.tmp â†’ .json)`. ESP reset mid-write
won't damage the previous file. SPIFFS requires removing the
destination before rename â€” hence the two-step.

File loaded on `events_service_init()`. If missing or unparseable, we
start with an empty list and write a fresh one.

## Recurrence rules â€” when does an event match today

`matches_today()` in [events_service.c](../components/events/events_service.c):

| Mode | Condition |
|---|---|
| `none` | Exact `year + month + day` match |
| `daily` | Always true |
| `weekly` | `weekday(base_date) == weekday(today)` â€” computed via `mktime()` + `localtime_r()` |
| `monthly` | `tm_mday == e->day` |
| `yearly` | `tm_mon+1 == e->month && tm_mday == e->day` (base year ignored) |

Intentionally not handled:

- **February 29** in `yearly` â€” in non-leap years `matches_today` returns
  false, the event simply doesn't fire that year.
- **Day > 28 in `monthly`** â€” in shorter months `matches_today` returns
  false, no rollover to the last day of the month.

## Scheduler

In [events_service.c](../components/events/events_service.c), started
from `events_service_init()`:

- Periodic `esp_timer`, interval `EVENTS_TICK_US = 30 * 1000 * 1000` (30 s).
- Calls `check_events()` plus an immediate first check (so we don't wait
  30 s after boot).

`check_events()` flow:

1. If `ntp_service_is_synced() == false` â†’ bail. Without time we don't
   fire anything.
2. `time(NULL)` â†’ `localtime_r(&now, &lt)`. Timezone from `TZ` env â€”
   set by [ntp_service](../components/services/ntp_service.c) at init.
3. Day rollover detection: if `s_last_day != lt.tm_mday` â†’ zero
   `s_fired_today[]`, store new day. Handles e.g. "event on April 20,
   fired before midnight, at 00:00:30 the next day we re-check from
   scratch".
4. Compute `now_min = lt.tm_hour * 60 + lt.tm_min`.
5. For each event: if `matches_today(e, &lt)` is true, compute
   `target = e->hour * 60 + e->minute`. If the **1-minute window**
   `now_min âˆˆ [target, target+1]` and `!fired_today` â†’ `fire_event(e)` +
   `fired_today = true`.

The 1-minute window with a 30 s tick guarantees every event is caught â€”
between any two ticks one will land inside the window.

**"Missed event" case**: ESP boots at 10:30, event scheduled for 08:00
the same day. Scheduler checks today's events â€” `target=480 < now_min=630`,
window already past, `fired_today` not set (but nothing fires either).
Next day flags reset, the event will fire again per recurrence rules.

## Triggering â€” `fire_event()`

Two actions:

**1. Audio.** Depends on type:
- `birthday` / `anniversary` â†’ `MELODY_BIRTHDAY` ("Happy Birthday" snippet)
- `reminder` / `nameday` â†’ `MELODY_REMINDER` (3Ã— short 880 Hz tone)
- `voice` â†’ `settings_set_volume(e->volume)` then `radio_play_notification()`
  plays the pre-rendered WAV from `/voice` on the SD card, interrupting and
  then restoring the current source. Missing clip/SD â†’ falls back to
  `MELODY_REMINDER`.
- `schedule` â†’ `settings_set_volume(e->volume)` forces the configured volume,
  then starts the source as if the user pressed play: empty `sound` â†’
  `radio_play_index(e->station)`; otherwise `stat()` on `SD_MOUNT_POINT + sound`
  picks file (`sd_player_play_path`) vs folder (`sd_player_play_folder`). No
  buzzer, no toast â€” `on_event_fired` navigates to `SCREEN_RADIO` / `SCREEN_SD`
  instead (see below). The stream/track keeps playing; stop it like any
  regular playback. Out-of-range `station` or missing SD â†’ logged warning,
  silent. The optional stop record has the same `playback_id`, empty `sound`
  and `station == EVENT_STATION_STOP` (-1); it calls `radio_stop()` (same as
  the user's Stop button), leaves volume untouched and does not navigate.
  The scheduler still sees two ordinary triggers, while the REST API and UI
  treat them as one range such as 07:00â€“08:00.

Buzzer patterns live in the shared melody registry
[components/buzzer/melodies.c](../components/buzzer/melodies.c) (also used by
the dashboard screensaver for threshold alerts); triggered from
[events_service.c](../components/events/events_service.c) via
`melody_play(MELODY_*)`. The buzzer module plays the pattern on its own
FreeRTOS task as `freq_hz, duration_ms` pairs.

**2. UI callback.** Invokes pointer `s_fire_cb` (if registered).
Dependency inversion: `events_service` doesn't know about UI, the UI
registers itself via `events_service_set_fire_cb()`. Translation to
`UI_EVT_EVENT_FIRED` happens in
[components/ui/ui_manager.c](../components/ui/ui_manager.c) function
`on_event_fired()` â€” that's where a `ui_event_t` with
`id/title/type_label/hour/minute` is built and pushed onto the UI queue.

## Corner indicator â€” event_indicator_widget

[components/ui/widgets/event_indicator_widget.c](../components/ui/widgets/event_indicator_widget.c)
â€” small label placed manually on selected screens (e.g.
[screen_home.c](../components/ui/screens/screen_home.c)) next to
`mode_indicator_widget`.

Display:
- `events_pending_today_count() == 0` â†’ hidden
- `== 1` â†’ just `ðŸ””`
- `>= 2` â†’ `ðŸ”” N`

Refreshes **on its own** via an internal `lv_timer` every 30 s.
Decision: we don't rely on a global `UI_EVT_CLOCK_TICK` (the enum exists
but nobody emits it in this project), nor on callbacks from
`events_service`. With a 30 s interval the counter shows up within 30 s
of adding an event from the web UI â€” acceptable.

Counter function in [events_service.c](../components/events/events_service.c):

```c
int events_pending_today_count(void);
```

Counts only events satisfying all of:
- `enabled == true`
- `!s_fired_today[i]`
- `matches_today(e, &lt)` true
- `target = hour*60 + minute >= now_min` (still ahead today)

So for an anniversary on April 20: the bell appears only between
midnight and the trigger time **on that specific day**. Outside April 20
â€” `matches_today` returns false, no bell.

## REST API

In [components/web/http_server.c](../components/web/http_server.c):

| Method | Path | Body / effect |
|---|---|---|
| `GET` | `/api/events` | Array of all events (JSON) |
| `POST` | `/api/events` | Body: event JSON (id optional). Returns the created object. |
| `POST` | `/api/events/playback` | Creates a start and optional linked stop in one save. `stop_enabled` controls whether `stop_hour` and `stop_minute` are required. |
| `PUT` | `/api/events/playback/{playback_id}` | Replaces the start and optional stop records of a playback schedule in one save. |
| `DELETE` | `/api/events/playback/{playback_id}` | Deletes the complete playback schedule in one save. |
| `PUT` | `/api/events/calendar` | Bulk "wipe & replace" of all `EV_CALENDAR` events. Body: JSON array (or `{"events":[...]}`). Returns `{"ok":true,"count":N}`. `[]` clears them all. |
| `PUT` | `/api/events/{id}` | Body: partial or full event JSON (partial patch). |
| `DELETE` | `/api/events/{id}` | Deletes the event |

> The `calendar` bulk route is registered **before** the `/api/events/*`
> wildcard. The playback schedule routes are also registered before the generic
> wildcard routes.

Validation in `event_validate()` â€” non-empty `title`, field ranges, type
in allowed enum. Bad fields â†’ HTTP 400 with the error text.

`PUT` with a single field works as a "patch" â€” `event_patch_from_json()`
overwrites only fields present in body. Used by the "Enabled" toggle in
the list: `fetch('/api/events/xyz', { method: 'PUT', body: '{"enabled":false}' })`.

## Web page

[spiffs_image/www/events.html](../spiffs_image/www/events.html) â€” single
page: form on top (add / edit) + event list below. JavaScript in
[events.js](../spiffs_image/www/events.js):

- `evLoad()` â€” fetch GET `/api/events`, render list
- `evSave()` â€” POST or PUT depending on `editingId`
- `evEdit(id)` â€” fills the form, switches to edit mode
- `evDelete(id)` â€” `confirm()` + DELETE
- `evToggleEnabled(id, on)` â€” PUT with one field

The Playback tab groups raw `schedule` records by `playback_id`. Stop time is
optional and its input is shown only after selecting **Stop playback at**. Creating, editing,
toggling or deleting a displayed range uses the playback endpoints, so a
partial schedule is never persisted. Playback recurrence is limited to one-time,
daily and weekly. When stop time is at or before start time, the stop record's
base date is advanced by one day (for example 22:00â€“06:00).
The date input is hidden for `daily`, where the scheduler ignores the base date;
for `weekly` the selected date defines the weekday.

On form reset the date defaults to today and time to now (helpers
`todayStr()` / `nowTimeStr()`) â€” fewer clicks for the common "add for
now" case.

## Calendar mirror (`EV_CALENDAR`)

`calendar` events are a **read-only projection of the phone calendar**, pushed
in bulk from the Android app's Calendar tab. The phone calendar is the source of
truth, so on the device these events:

- **never fire** â€” `check_events()` skips `EV_CALENDAR` (no buzzer, no toast,
  they never reach `fire_event()`).
- are **excluded** from `events_pending_today_count()` (the ðŸ”” counter).
- exist only to feed the Home-screen calendar widget (below).

Because the source of truth lives off-device, they're managed as a group via a
**wipe & replace** bulk sync rather than per-event CRUD â€” no id tracking needed,
the type itself is the tag:

| Function ([events_service.c](../components/events/events_service.c)) | Role |
|---|---|
| `events_replace_calendar(arr, n)` | Drops every existing `EV_CALENDAR`, then appends `arr[0..n)` (type forced, id generated when empty), one SPIFFS write. Non-calendar events untouched. |
| `events_calendar_current(out)` | Copies the next upcoming `EV_CALENDAR` for today (soonest time not yet passed) into `out`; returns false if none / NTP unsynced. A **copy**, not a pointer, so a concurrent bulk replace can't pull data out from under the LVGL task. |

The `PUT /api/events/calendar` endpoint parses the array, forces the type and
validates each item (malformed entries skipped), then calls
`events_replace_calendar()`. Sending `[]` clears all calendar events (the app's
"Clear" button).

## Known limitations

- `fired_today` flags live in RAM. ESP reset before midnight â†’ an event
  fired earlier the same day will fire again after boot if it lands in
  the time window. Tradeoff â€” alternative is persisting the flags to
  SPIFFS on every fire, which we don't want.
- Timezone. Changing TZ at runtime via `ntp_service_reconfigure()` does
  **not** call `events_service_on_tz_changed()` automatically â€” no hook.
  The hook is prepared in the API but intentionally not wired to avoid
  a circular dependency. In practice: set TZ once on first run.
  `settings.html` has a warning about this.
- Buzzer patterns are fixed â€” not configurable per event.
- Maximum 200 events (user-managed + calendar-mirror combined).
